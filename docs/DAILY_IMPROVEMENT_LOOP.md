# Щоденний цикл покращення для особистого агента OpenClaw

## Мета
Побудувати безпечний щоденний процес, який 1 раз на день:
- збирає новинки у сфері агентів, памʼяті, моделей, браузерних агентів, голосу, автоматизації, медицини та бізнес-інструментів;
- оцінює, чи підходять вони для самого агента OpenClaw;
- окремо оцінює, чи підходять їхні найкращі частини для інших моїх проєктів;
- зберігає результат у структурований markdown-звіт;
- ніколи не змінює код автоматично без окремого дозволу.

---

## Головний принцип
Система повинна бути не "самопокращенням без контролю", а "щоденним аналітичним контуром".

Правильна модель:
1. знайти;
2. відфільтрувати шум;
3. оцінити цінність;
4. витягнути корисні частини;
5. запропонувати конкретні покращення;
6. зберегти у звіт;
7. за бажанням створити окремий список задач.

---

## Чого система НЕ робить
- не змінює production-код автоматично;
- не мерджить pull request автоматично;
- не переписує архітектуру без ревʼю;
- не вірить маркетингу без оцінки зрілості;
- не зберігає все підряд без пріоритизації.

---

## Щоденний конвеєр

### Крок 1. Збір джерел
Бажані джерела:
- GitHub Releases / Changelog важливих проєктів;
- блоги Anthropic, OpenAI, Meta, Google, xAI, Perplexity;
- новини про OpenClaw, памʼять, агентів, браузерні агенти, голос, automation, clinical AI;
- вибрані RSS-стрічки;
- список ручних джерел, яким я довіряю.

### Крок 2. Первинний фільтр
Залишати тільки те, що має відношення хоча б до одного з напрямків:
- агентна архітектура;
- memory governance;
- retrieval;
- browser agent;
- voice agent;
- deterministic workflow;
- observability / audit;
- medical logic;
- business metrics;
- lower cost / local inference;
- UX для асистента.

### Крок 3. Оцінка кожної новинки
Для кожної знайденої новинки рахувати бали за шкалою 0-5:

- **Релевантність для OpenClaw**
- **Релевантність для AIRYS та інших проєктів**
- **Практична користь**
- **Складність інтеграції**
- **Ризик / нестабільність**
- **Зрілість рішення**
- **Наскільки це можна використати частково, а не цілком**

### Крок 4. Класифікація рішення
Кожна новинка має потрапити лише в одну з категорій:

1. **Взяти майже повністю**
2. **Взяти окремі сильні частини**
3. **Лише спостерігати**
4. **Поки ігнорувати**

### Крок 5. Витягування найкращих частин
Для категорії "Взяти окремі сильні частини" система повинна відповісти:
- що саме там сильне;
- у який мій проєкт це підходить;
- чи це про архітектуру, памʼять, UX, агентність, голос, аналітику або бізнес-метрики;
- який мінімальний спосіб це перевірити;
- який ризик, якщо копіювати без адаптації.

### Крок 6. Формування щоденного висновку
Наприкінці дня система повинна формувати 3 короткі блоки:

1. **Що реально варте уваги сьогодні**
2. **Що дати в backlog**
3. **Що не варто чіпати зараз**

### Крок 7. Створення артефактів
Щодня створювати:
- один файл звіту;
- за потреби оновлення watchlist-файлу;
- за потреби оновлення backlog-файлу.

## Найкращий режим запуску

### Мінімальний варіант
1 раз на день запускати збір і аналіз.

Час:
- щодня о 07:30 або 08:00 за локальним часом.

Вихід:
- новий markdown-звіт у `/docs/daily-research/`.

### Production-підхід
1. окремий збирач джерел;
2. окремий етап дедуплікації;
3. окремий етап оцінки;
4. окремий етап збереження;
5. окремий weekly review раз на тиждень.

---

## Логіка прийняття рішень

### Якщо новинка дає повноцінну перевагу
Тільки тоді позначати як "взяти майже повністю".

### Якщо новинка сильна лише частково
Позначати як "взяти окремі сильні частини" і обовʼязково виписувати:
- яку саме частину;
- як її адаптувати;
- чому не брати все рішення повністю.

### Якщо новинка лише шумить
Позначати як "ігнорувати" навіть якщо вона популярна.

---

## Дуже важливе правило безпеки
Будь-яке покращення проходить через 3 стани:

1. **Ідея**
2. **Перевірка у markdown-звіті**
3. **Окреме рішення про імплементацію**

Система аналізу не повинна автоматично ставати системою самозміни.

## Рекомендована реалізація

### Варіант 1. Найпростіший
- cron раз на день;
- Python-скрипт або n8n;
- запис markdown-файлу;
- без автоматичних змін коду.

### Варіант 2. Правильний для росту
- джерела + парсер;
- scoring-модуль;
- генератор звіту;
- backlog updater;
- weekly review;
- окремий manual gate перед будь-якою реальною зміною коду.

---

## Практичне рішення для старту
Почати з такого мінімуму:

1. один щоденний запуск;
2. один звіт у markdown;
3. чотири категорії вердикту;
4. окремий backlog;
5. жодного автопереписування коду.

Це дасть користь майже одразу і не зламає систему.
